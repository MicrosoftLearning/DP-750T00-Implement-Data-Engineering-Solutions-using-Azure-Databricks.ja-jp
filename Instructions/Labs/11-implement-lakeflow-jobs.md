# ラボ 11: Azure Databricks を使用して Lakeflow ジョブを実装する

## はじめに

あなたは **TelConnect** のデータ エンジニアです。同社は、毎日何百万件もの通話、SMS メッセージ、データ セッションを管理する地域の電気通信プロバイダーです。 各イベントによって、ネットワーク アクティビティの生信号である **詳細な通話の記録 (CDR)** が生成されます。 このデータは、運用チームがネットワーク パフォーマンス レポートと容量計画に使用する前に、多層パイプラインを確実に通過する必要があります。

現在の課題は、**Lakeflow ジョブ**を使用してそのパイプラインを自動化することです。 次の処理を行います。

- CDR データを処理する事前構築済みのパラメーター化された **medallion アーキテクチャ** ノートブック (ブロンズ → シルバー → ゴールド) を実行する
- Azure Databricks UI で、調整済みのタスクとしてノートブックを実行する **Lakeflow ジョブ** を作成する
- 動的トリガー値を使用して、`processing_date` **ジョブ パラメーター**をノートブックに渡す
- **タスクの依存関係**を持つ 2 番目のダウンストリーム タスクを追加する
- 新しい CDR ファイルがボリュームに到着したときに発生する **イベント ベースのファイル到着トリガー** を構成する
- さらに、**夜間の cron スケジュール**を追加して、ジョブが毎晩自動的に実行されるようにする
- **失敗通知**を設定して、ネットワーク運用チームがただちにアラートを受け取るようにする
- 一時的なクラウドストレージ エラーを処理するように**自動再試行ポリシー**を構成する

## パート 1: パイプライン ノートブックを実行する

ノートブックには、ブロンズ → シルバー → ゴールド パイプラインを構築する 5 つの演習が含まれています。 すべてのセルが用意されています。パート 2 で調整するデータ環境を設定するために、それらのセルを実行してください。

### ノートブックをインポートする

1. Databricks ワークスペースで、左側のサイド バーにある **[ワークスペース]** をクリックします。
2. このラボを保存するフォルダーに移動するか、そのフォルダーを作成します。
3. **[⋮]** (ケバブ メニュー) をクリックするか、フォルダーを右クリックして、**[インポート]** を選択します。
4. **[URL]** を選択し、URL (`https://raw.githubusercontent.com/MicrosoftLearning/DP-750T00-Implement-Data-Engineering-Solutions-using-Azure-Databricks/refs/heads/main/Allfiles/11-implement-lakeflow-jobs.ipynb`) を入力し、**[インポート]** をクリックします。
5. インポートしたノートブックを開き、上部にあるコンピューティング セレクターで **[サーバーレス]** コンピューティングを選択します。

### ノートブックを実行する

1. **演習 1** のセルを実行します。これにより、ブロンズ、シルバー、ゴールドのスキーマを含む Unity カタログ **telconnect_lab** が作成されます。
2. **[演習 2]** で、最初のセルを実行して **raw_uploads** ボリュームを作成します。**telecom_cdrs.csv** が GitHub からボリュームに自動的にダウンロードされます。
3. **演習 5** まで残りのセルをすべて実行します。

> **ヒント:** 各セルには ✅ のマークが付けられています。 各コード セルの上にあるマークダウンを読んで、そのステップの実行内容を理解してから実行してください。この文脈は、パート 2 で Lakeflow ジョブを構成するときに役立ちます。

これらの演習は、このラボの中核です。 Azure Databricks の **[ジョブとパイプライン]** UI を使用して、トリガー、スケジュール、通知、再試行ポリシーを使用する運用環境レベルの自動化されたジョブに、先ほど実行したノートブックを接続します。

### 演習 1: CDR パイプライン ノートブックを使用して Lakeflow ジョブを作成する

#### ジョブを作成する

1. 左側のサイドバーで、**[ジョブとパイプライン]** をクリックします。
2. **[ジョブの作成]** をクリックします。
3. 既定のカタログ名を `TelConnect CDR Pipeline` に置き換えます。

#### 最初のタスクを構成する

4. タスク エディターで、タスク名 `process_cdrs`を入力します。
5. **[種類]** ドロップダウンで、**[ノートブック]** を選択します。
6. **[ソース]** ドロップダウンで、**[ワークスペース]** を選択し、インポートしたノートブック (*11-implement-lakeflow-jobs*) のパスを参照します。
7. **[コンピューティング]** セレクターで、**[サーバーレス]** を選択します。
8. **[タスクの保存]** をクリックします。

#### ジョブ パラメーターを追加する

9. 右側の **[ジョブの詳細]** パネルで、**[パラメーター]** セクションまでスクロールし、**[追加]** をクリックします。
10. キーを `processing_date` に設定し、値を `{{job.trigger.time.iso_date}}` に設定します。

   > この動的な値により、ジョブがトリガーされた日付が挿入され、ノートブックのウィジェットの既定値と一致します。

11. パラメーターの **[保存]** をクリックします。

#### 確認

12. タスク グラフで、**process_cdrs** が単一ノードとして表示されていることを確認します。
13. **[今すぐ実行]** をクリックして、ジョブを手動でテストします。
14. **[実行]** タブで、実行が完了するのを待ち、すべてのノートブック セルが正常に実行されたことを確認します。

---

### 演習 2: ダウンストリーム通知タスクを追加する

実際のパイプラインでは、CDR 処理タスクの完了後に、2 番目のタスクで概要レポートを送信することが必要な場合があります。 ここでは、ゴールド テーブルから読み取る短い SQL タスクを使用して、それをシミュレートします。

1. **[TelConnect CDR Pipeline]** ジョブ エディターで、**process_cdrs** の下部にある **[+ タスクの追加]** をクリックします。
2. タスクに `summarize_gold` という名前を付けます。
3. **[種類]** ドロップダウンで、**[SQL]** を選択します。
4. **[SQL]** フィールドに次のように入力します。

   ```sql
   SELECT region, network_type, total_calls, drop_rate_pct
   FROM telconnect_lab.gold.network_summary
   ORDER BY drop_rate_pct DESC;
   ```

5. **[SQL ウェアハウス]** セレクターで、使用可能なサーバーレス SQL ウェアハウスを選択します。
6. **[依存先]** が既に **process_cdrs** に設定されており、**Run if** 条件が **[すべて成功]** であることを確認します。
7. **[タスクの保存]** をクリックします。

#### DAG を検証する

8. DAG ビューを確認します。 次のような結果が表示されます。
   ```
   process_cdrs → summarize_gold
   ```

### 演習 3: ファイル到着トリガーを構成する

TelConnect は、1 日を通して不規則な間隔で新しい CDR ファイルをボリュームにドロップします。 固定スケジュールでポーリングするのではなく、新しいファイルが到着するたびに自動的に起動するようにジョブを構成します。

1. **[ジョブの詳細]** パネルの **[スケジュールとトリガー]** で、**[トリガーの追加]** をクリックします。
2. **[トリガーの種類]** ドロップダウンで、**[ファイル到着]** を選択します。
3. **[保存場所]** フィールドに、次のように入力します。

   ```
   /Volumes/telconnect_lab/bronze/raw_uploads/
   ```

4. **[詳細オプション]** で、**[最後の変更後の待機時間]** を `60` 秒に設定します。

   > これにより、急速な複数のファイル到着が 1 回のジョブ実行でバッチ処理されます。これは、複数の CDR バッチが短時間に連続して到着する場合に便利です。

5. **[保存]** をクリックします。

#### トリガーを一時停止する

次の演習でスケジュールも追加するので、ラボ中に重複実行を避けるために、ファイル到着トリガーを一時停止します。

6. **[スケジュールとトリガー]** で、ファイル到着トリガーを見つけて、**[一時停止]** をクリックします。

> **注:** 運用環境では、両方をアクティブのままにします。ファイル到着トリガーは日中の更新を処理し、スケジュールは信頼性の高い夜間のキャッチオール実行を提供します。

---

### 演習 4: 夜間スケジュールを追加する

TelConnect の運用チームは、午前のシフトが始まる前にダッシュボードが最新の状態になるように、ゴールド サマリー テーブルが毎夜 2 時 (CET) に更新されることを期待しています。

1. **[ジョブの詳細]** パネルで、**[トリガーの追加]** を選択します。
2. **[トリガーの種類]** を **[スケジュール済み]** に設定します。
3. **[スケジュールの種類]** を **[詳細]** に設定します。
4. **[Cron 式]** フィールドに次のように入力します。

   ```
   0 0 2 * * ?
   ```

   > この式は、どの曜日でも、毎月、毎日、2 時 0 分 0 秒に実行することを意味します。

5. **[タイム ゾーン]** を **[ヨーロッパ/アムステルダム]** に設定します。
6. **[保存]** をクリックします。

#### cron 式を確認する

7. 保存した後、人間が判読できるスケジュール プレビューを確認します。 "毎日アムステルダム時間の午前 2 時" と表示されます。**

---

### 演習 5: 失敗通知を構成する

CDR パイプラインが失敗した場合、ネットワーク運用チームはただちにアラートを受け取る必要があります。実行が失敗すると、障害検出とネットワーク SLA レポートが遅れる可能性があります。

#### ジョブ レベルの失敗通知を追加する

1. **[ジョブの詳細]** パネルで、**[ジョブ通知]** までスクロールし、**[通知の編集]** をクリックします。
2. **[通知の追加]** をクリックします。
3. **[メール アドレス]** を選択し、自分のメール アドレスを入力します。
4. イベントの種類として **[失敗]** を選択します。
5. 必要に応じて、**[期間の警告]** を選択し、しきい値を `30` 分に設定します。
6. **[保存]** をクリックします。

#### タスク レベルの再試行通知を追加する

7. タスク グラフで **[process_cdrs]** タスクを選択して、その構成を開きます。
8. タスク パネルで、**[通知]** までスクロールし、**[追加]** をクリック。
9. **[メールアドレス]** (同じアドレス)、**[失敗]** の順に選択し、**[最後の再試行まで通知をミュートする]** をオンにします。

   > これにより、タスクの再試行時の通知スパムが防止されます。 再試行がすべて失敗した場合にのみ、アラートを受け取ります。

10. **[タスクの保存]** をクリックします。

---

### 演習 6: 自動再試行を構成する

CDR ファイルはクラウド ストレージから読み込まれるため、一時的なネットワーク エラーやストレージのスロットリングが、有効期間の短いタスクが失敗する一般的な原因となります。 再試行を設定すると、パイプラインは手動による介入なしに復旧します。

#### process_cdrs のタスク レベルの再試行を構成する

1. タスク グラフで **[process_cdrs]** タスクを選択します。
2. タスク構成パネルで、**[再試行]** の横にある **[+ 追加]** をクリックします。
3. **[再試行の回数]** を `2` に設定します。
4. **[再試行の間隔]** を `60` 秒に設定します。
5. **[タスクの保存]** をクリックします。

#### タイムアウトしきい値を構成する

6. 同じタスク構成で、**[メトリックしきい値]** をクリックします。
7. **[メトリック]** を **[実行期間]** に設定します。
8. **[警告]** を `20` 分に設定します。
9. **[タイムアウト]** を `30` 分に設定します。
10. **[タスクの保存]** をクリックします。

> ノートブックがハングした場合 (たとえば、大きな CDR バックログでクエリが停止した場合)、タイムアウトによって、タスクは失敗しても安全に停止することが保証されるので、タスクが無限に実行されるのではなく、再試行メカニズムを作動させることができます。

---

### 演習 7: 完全なジョブを実行して監視する

1. **[TelConnect CDR Pipeline]** ジョブ エディターで、**[今すぐ実行]** をクリックします。
2. **[実行]** タブに移動し、アクティブな実行をクリックして実行の詳細を開きます。
3. 両方のタスク (process_cdrs と summarize_gold) が順番に実行されているのを確認します。
4. 実行タイムラインで **[process_cdrs]** をクリックして、次の情報を表示します。
   - ノートブックの出力 (各セルの結果)
   - 渡されたジョブ パラメーター値 (processing_date)
   - dbutils.jobs.taskValues.set() によって設定されたタスク値
5. 両方のタスクが完了したら、実行の状態が **[成功]** であることを確認します。

## まとめ

このラボでは、次のことを行いました。

- Unity Catalog での TelConnect CDR データ用 **medallion アーキテクチャ** パイプライン (ブロンズ → シルバー → ゴールド) の実行
- 2 つのタスクとシーケンシャル タスクの依存関係を含む **Lakeflow ジョブ** の作成
- トリガー時間に関連付けられた**動的ジョブ パラメーター**としての processing_date の受け渡し
- 新しい CDR ファイルがボリュームに到着したときに起動する**ファイル到着トリガー**の構成
- 毎日の更新を確実に行うための**夜間の cron スケジュール** (02:00 CET) の追加
- アラート ノイズを低減するために再試行アラートをミュートした**失敗通知**の設定
- **タスク レベルの再試行** (2 回の試行、60 秒間隔) と、30 分のタイムアウトしきい値の構成
