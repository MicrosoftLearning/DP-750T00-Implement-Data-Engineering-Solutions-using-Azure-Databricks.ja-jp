# ラボ 13: Azure Databricks でワークロードを監視、トラブルシューティング、最適化する

## はじめに

以前は数秒で完了したクエリに数分かかるようになりました。 クラスターのリソースはビジー状態に見えますが、ほとんどのタスクはほぼ瞬時に完了し、時間がかかっているのはほんの一部のタスクだけです。 これらは**データ スキュー**の特徴であり、多くの場合、**過剰なシャッフル**を伴います。これにより、Spark は、ネットワークを介してステージ間で大量のデータを移動させる必要があります。

このラボでは、両方の問題を意図的に再現し、Spark UI 上でそれらがどのように表示されるかを正確に観察した後、対象となる修正を適用して改善を確認します。

| 部分 | トピック | 場所 |
|------|-------|--------|
| **パート 1** | 環境を設定してデータを生成する | ノートブック |
| **パート 2** | データスキューを明らかにし、Spark UI を検査する | ノートブックとラボの手順 |
| **パート 3** | データ スキューを修正する | ノートブック |
| **パート 4** | 過剰なシャッフルを明らかにし、Spark UI を検査する | ノートブックとラボの手順 |
| **パート 5** | シャッフルのオーバーヘッドを削減する | ノートブック |

## コンピューティング クラスターを作成する

このラボでは、**クラシック コンピューティング クラスター** (サーバーレスではない) が必要です。 アダプティブ クエリ実行の無効化と再有効化に使用される一部の Spark 構成オプション (*spark.databricks.optimizer.adaptive.enabled* など) は、クラシック クラスターでのみ使用できます。

> ⚠️**Photon を無効にする必要があります。** Photon は、Databricks のベクター化クエリ エンジンです。 これを有効にすると、シャッフルとスキューの処理が自動的に最適化されるため、このラボで実証しようとしているパフォーマンスの問題が隠れてしまいます。 Photon がアクティブな場合、Spark UI で偏りのあるタスク分布や高い値のシャッフル メトリックは表示されません。

1. Databricks ワークスペースで、左側のサイド バーにある **[コンピューティング]** をクリックします。
2. **[コンピューティングの作成]** をクリックします。
3. 次の最小設定でクラスターを構成します。

   | 設定 | 値 |
   |---------|-------|
   | **[Cluster name](クラスター名)** | `perf-lab` |
   | **ポリシー** | 無制限 |
   | **単一ノード** | Enabled |
   | **アクセス モード** | 1 人のユーザー |
   | **Databricks Runtime** | 最新の LTS (非 ML) - **必ず非 Photon ランタイムを選択してください** |
   | **ノードの種類** | 使用可能な最小のノード タイプ (`Standard_DS3_v2` または同等のものなど) を選択します |
   | **自動終了** | 30 分 |

4. **[詳細オプション]** を展開し、**[Spark]** タブをクリックします。**[Spark 構成]** ボックスに次の行を追加して、Photon を明示的に無効にします。

   ```
   spark.databricks.photon.enabled false
   ```

5. **[コンピューティングの作成]** をクリックし、クラスターが **[実行中]** 状態になるまで待ってから続行します。

## パート 2: Spark UI でデータ スキューを調査する

ノートブックの **演習 2** でセルを実行した後、ここに戻り、Spark UI で次の手順に従って、偏りのある実行を確認します。

### Spark UI を開く

1. Databricks ワークスペースで、左側のサイドバーにある **[コンピューティング]** をクリックし、**[perf-lab]** クラスターをクリックします。 クラスターの詳細ページで、**[Spark UI]** タブをクリックします。Spark UI が新しいブラウザー タブで開きます。
2. **[ジョブ]** ページに移動します。このページには、このセッション中にトリガーされたすべての Spark ジョブが一覧表示されます。

### 偏りのある集計ジョブを特定する

3. 最後に完了したジョブを見つけます。 タスク 2.1 の **[groupBy 集計]** とタスク 2.2 の **[並べ替えマージ結合]** が上部付近に表示されます (新しい順)。
4. 集計ジョブをクリックします (*aggregate* または *hashAggregate* に関連するステージが参照されます)。
5. ジョブの詳細ページで、**[ステージ]** の一覧を確認します。 シャッフルを実行したステージをクリックします。これは、通常、タスクが最も多く、期間が最も長いステージです。
6. そのステージのタスクの **[サマリー メトリック]** テーブルまで下にスクロールします。 **[期間]** 行にフォーカスします。
   - **[最小]**、**[25 パーセンタイル]**、**[中央]**、**[75 パーセンタイル]**、**[最大]** の各値に注意してください。
   - **[最大]** が **[75 パーセンタイル]** より 50% 以上高い場合、データ スキューが発生します。つまり、少数のタスクでデータの大部分が処理されています。
7. **[タスク]** セクションまでさらに下にスクロールします。 一覧を **[期間]** (降順) で並べ替えます。 1 つまたは 2 つのタスクの期間がどのようにして残りのタスクよりもはるかに長くなるかを確認します。

### 偏りのある結合ジョブを特定する

8. Spark UI の左側のナビゲーションで **[ジョブ]** をクリックして、ジョブの一覧に戻ります。
9. タスク 2.2 の並べ替えマージ結合ジョブをクリックし、分析を繰り返します。 同じパターンを探します。大多数のタスクよりも期間がはるかに長いタスクがいくつかあります。
10. 調査が完了したら、ノートブックに戻り、**演習 3** に進みます。

## パート 4: Spark UI で過剰なシャッフルを調査する

ノートブックの **演習 4** でセルを実行した後、ここに戻ってシャッフル メトリックを調査します。

### シャッフルが多いステージを見つける

1. [Spark UI] タブに戻ります (または、ノートブックのツール バーから再度開きます)。
2. 左側のナビゲーションで **[ステージ]** タブをクリックして、すべてのジョブのすべてのステージを表示します。
3. **[シャッフル読み取り]** 列と **[シャッフル書き込み]** 列を確認します。 演習 4 によってトリガーされるステージでは、両方の列で大幅なデータ移動が示されています。多くの場合、実際の入力データのサイズをはるかに上回る移動です。

### DAG 内の Exchange ノードの数をカウントする

4. シャッフル値が高いステージをクリックして、詳細ページを開きます。
5. ステージの詳細ページの上部で、**[DAG の視覚化]** を展開します。 **Exchange** というラベルが付いたノードを探します。各ノードは、クラスター ネットワーク全体のデータの完全なシャッフルを表します。
6. 演習 4 のジョブのステージで表示される Exchange ノードの合計数をカウントします。 これを、groupBy → 結合→並べ替えパイプラインに実際に必要なシャッフルの最小数と比較します。

### シャッフル読み取りと入力サイズの比較

7. ステージの詳細ページを開いたまま、**[サマリー メトリック]** セクションまでスクロールします。 **[入力サイズ/レコード]** と **[シャッフル読み取りサイズ/レコード]** を比較します。 シャッフル読み取りが元の入力よりも大幅に大きい場合、パイプラインは必要以上に多くのネットワーク I/O を実行しています。
8. 調査が完了したら、ノートブックに戻り、**演習 5** に進みます。
